## 3. 「隔日沖大戶」演算法

### 3.1 業界共識定義

> 同一分點(券商 + 分行)在某檔股票:
> - 第 D 日大買 ≥ N 張(實務 N = 100–500 視個股股本)
> - 第 D+1 日大賣 ≥ M%(實務 M = 70–100% 即幾乎全部出清)
> - 上述 pattern 在過去 K 個月內重複 ≥ R 次(實務 K = 3–6, R = 3–10)
> - 滿足以上 → 標記該分點為「該股的隔日沖客戶」或「全市場隔日沖客戶」

市場熟知名單(社群長期觀察):**凱基-台北、富邦-敦南、群益-松江、元大-永和、康和-延平**等。但要注意這只是「歷史曾經是」,不代表現在還是 — 主力身分會輪替。

### 3.2 算法

```
偽碼:
for 每個分點 b in 全市場分點:
  for 每個歷史交易日 D in 過去 90 天:
    買入張數 = broker_daily[b, D]['buy']
    賣出張數 = broker_daily[b, D+1]['sell']  # 隔日
    if 買入張數 >= 100 and 賣出張數 / 買入張數 >= 0.7:
      隔日沖事件 += 1
  if 隔日沖事件 >= 5:  # 90 天內 5 次
    標記為「隔日沖大戶」
```

複雜度:O(分點數 × 個股數 × 90 天) = **O(2000 × 2400 × 90) ≈ 4 億次 join 操作**(naive)。
若用 SQL 做 self-join 上 index 預估 5–30 分鐘 / 全量重算。可接受。

### 3.3 歷史資料需求量

- **最低 60 天**:才能看出 pattern 統計顯著性(< 30 天容易誤判)
- **理想 90–180 天**:涵蓋至少一個季報週期的主力輪動

### 3.4 用既有 ML 還是純規則?

- 純規則(threshold-based)即可:這類「行為 fingerprint」是**規則導向**,RandomForest 沒邊際價值
- 既有 `ml_predictor` 可保留作為「跟單後預測 5/10 日報酬」用途(見第 5 節)

## 4. 「波段大戶」演算法

### 4.1 業界共識定義

> 同一分點在某檔:
> - 第 D 日大買 ≥ N 張
> - 第 D+1 ~ D+5 日**沒有大量反向**(賣出 < 10% 買入量)
> - 第 D+6 ~ D+20 日繼續加碼或不動(累積買賣比 ≥ 1.2)
> - 通常是法人代操、自營部位、家族辦公室、中期主力

### 4.2 算法

```
偽碼:
for 每個分點 b:
  for 每個股票 s:
    買入累積 = sum(broker_daily[b, s, D-20..D]['buy'])
    賣出累積 = sum(broker_daily[b, s, D-20..D]['sell'])
    if 買入累積 >= 500 and 買入累積 / max(賣出累積, 1) >= 1.5:
      標記為「波段大戶持倉中」
```

需要的歷史資料:**至少 60 天**(看完整一個 20 日波段 + 60 日趨勢)。
複雜度:O(分點 × 個股 × 20 日 rolling) ≈ 1 億次運算,可接受。

### 4.3 與隔日沖的互斥性

實務上同一分點對同一檔股票通常**只會是一種角色**(波段或隔日沖),所以 (b) 和 (c) 算完後可以做互斥分類,降低後續跟買的雜訊。

## 5. 跟買分析方法

### 5.1 回測指標

對「隔日沖大戶買入事件」與「波段大戶買入事件」分別計算:
- 後 1 日報酬中位數 / 平均 / 勝率
- 後 5 日報酬中位數 / 平均 / 勝率
- 後 10 日報酬中位數 / 平均 / 勝率
- 後 20 日報酬中位數 / 平均 / 勝率
- 最大回撤(進場後到出場前最低點)
- 配對 t-test 檢定是否顯著優於市場平均

### 5.2 與既有 ML pipeline 整合

**方案 A(分開)**:獨立 backtest 模組,輸出靜態勝率表 → Pick Card 顯示「跟單預期勝率 58% / 平均報酬 +2.3%」
**方案 B(整合)**:把「分點訊號」作為 ML 特徵塞入 `ml_predictor.py`(現有 11 特徵 → 12–14 特徵),重訓 → 用 ML 輸出機率

軍師建議:**A 先做(快、可解釋),B 留待 A 驗證有 alpha 後再做**。

### 5.3 訊號 → 推薦清單流程

```
今天盤後 21:00:
  1. 從 broker_daily 拉今日全市場買進事件
  2. join 上「隔日沖 / 波段」分類表
  3. 按分點品質排序(歷史勝率高的優先)
  4. 輸出 Top N 「大哥今日買入」清單到 Streamlit
  5. (可選)Telegram 推播 Top 5
```

## 6. 資料量 + 成本估算

### 6.1 資料量(假設拿到全市場分點明細)

- 台股實際:**約 70 家券商總部,旗下約 1000–1100 個分公司/分點**(不是 130 × 1000;130 是大致券商家數含期貨,證券單位約 70)
- 上市櫃股票數:~2400 檔
- 平均每檔每日有交易紀錄的分點:~50–150 個(熱門股 500+,冷門股 < 10)

**每日新增 row 數估算**:
- 保守:2400 檔 × 100 分點 = **24 萬 row / 日**
- 樂觀(只記有交易):**5–15 萬 row / 日**

**90 天 backfill**:90 × 15 萬 = **1350 萬列**
**SQLite 儲存空間**:每列約 50–80 bytes(stock_id, date, broker, buy, sell) → **~1 GB**

### 6.2 與既有資料集對比

| 資料集 | 累積 row 數(估) | DB 大小 |
|---|---|---|
| `daily_prices`(2400 檔 × 250 日 × 5 年) | 300 萬 | ~150 MB |
| `institutional`(70 檔 × 250 日) | 1.7 萬 | < 5 MB |
| `holders_distribution`(2400 檔 × 52 週 × 17 級) | 200 萬 | ~80 MB |
| **`broker_daily`(新)** | **1350 萬 / 90 天** | **~1 GB** |

**結論**:`broker_daily` 一張表就**比現有所有表加總還大 5 倍**。SQLite 還能跑(理論上限 281 TB),但:
- index 重建時間會很長(初次 backfill 30+ 分鐘)
- GitHub Actions free tier 單個 job runner storage 限制 14 GB,沒問題;但 6 小時超時上限要留意
- repo 不能 commit 這麼大的 DB(GitHub 單檔 100 MB 上限),要走 release artifact 或外部儲存

### 6.3 跑批耗時估算

| 任務 | 估時 |
|---|---|
| 抓全市場 1 日分點(假設用 GitHub repo 包裝) | 5–15 分鐘 |
| 跑 90 天 backfill | 5–15 小時(GitHub Actions 要切多次 job) |
| 每日 incremental 跑「隔日沖 / 波段」分類 | 3–10 分鐘 |
| 跑跟買回測 | 5–20 分鐘 |

**GitHub Actions 月配額**(free tier 2000 分鐘 / 月):
- 既有 daily_fetch + weekly_shareholder + retrain_ml ≈ 30–60 分鐘 / 日 = ~1500 / 月
- 加上 broker daily ≈ 10 分鐘 / 日 = +300 / 月 → **逼近 free tier 上限**

## 7. 法律 / 合規(重要,主公必看)

### 7.1 資料 redistribute

| 來源 | 可否儲存 | 可否 redistribute | 可否商用 |
|---|---|---|---|
| TWSE bsr 原始 HTML | ✅ | ⚠️ 需註明來源 | ⚠️ |
| TWSE 經處理 derived signal(只列「凱基-台北買 X 張」) | ✅ | ✅ | ✅ |
| Goodinfo / wantgoo 爬蟲資料 | ❌ ToS | ❌ | ❌ |
| GitHub repo 二手資料 | 看授權 | 看授權 | 看授權 |

**個人專案 + 不對外發行**:基本上 OK。但這個專案的 GitHub repo 是 **public**,如果把 `broker_daily` 表 commit 進去,等於是 public redistribution,有灰色地帶。
**建議**:`broker_daily` 表加進 `.gitignore`,只 commit derived signals(分類結果 / 跟買清單)。

### 7.2 ⚠️ **金管會「薦股」風險(必讀)**

**這條最重要,主公一定要看**:

台灣《證券投資顧問事業管理規則》規定:
- 對「**特定** 個股」做出「**買進 / 賣出 / 持有**」的具體建議,屬於「投顧業務」,須有金管會核發的投顧執照
- 個人 / 非投顧業者公開推薦個股,可能觸法,輕則行政罰,重則刑事(如造成投資人損失)

**「大哥帶你飛」這個功能的本質就是個股薦股**:列出「今天大哥買了哪些股,你要不要跟」就是明確的買進建議。

**緩解措施**(必須做):
1. **限私人使用**:Streamlit 不可公開部署,只能本機 / VPN 內網
2. **免責聲明**:UI 上常駐顯示「僅供個人投資決策參考,非投資建議,本工具不對使用結果負任何責任」
3. **不對外推播**:Telegram 推播僅推給主公本人,不可建群組
4. **不公開 repo 中放跟買清單原始檔**

對比:既有「短線」「長線」strategy 也是個股推薦,但用的是公開技術指標 + 籌碼面**統計**訊號,不是直接「跟某個分點」,法律灰色地帶較淺。「大哥帶你飛」**直接點名分點 + 推薦跟買**,風險顯著高一階。

> 📌 **軍師強烈建議**:這個功能名字、UI 文案、推播文字都要去掉「跟單」「跟買」「大哥」「主力」等詞彙,改成中性敘述如「分點集中度高的個股」「籌碼異動」,降低法律風險。

## 9. MVP 三方案 + 不做方案

### 方案 A:純爬 TWSE bsr(免費但 CAPTCHA)

- **範圍**:只追主公 watchlist N 檔(~50 檔)
- **資料**:TWSE bsr 個股當日進出
- **CAPTCHA 處理**:
  - 自動 OCR(`pytesseract` + 預處理)準確率 60–80%,失敗 retry
  - 或半自動:每日早上人工解 5–10 次驗證碼解鎖 cookies
- **估時**:
  - CAPTCHA OCR pipeline:**1–2 週**(踩坑、調圖像處理參數)
  - SQLite schema + fetcher + cron:**3–5 天**
  - 隔日沖 / 波段大戶分類:**3–5 天**
